01 Context

The goal: an assistant that can chat freely, but cannot move money without verification

Required

  • Answer general portfolio queries (balances, accounts)
  • State-aware verification before sensitive actions
  • Production observability design
  • Automated evaluation with ADK evalsets

Why it is hard

An LLM is non-deterministic. The challenge is not calling an API — it is guaranteeing the agent never transfers money without verification, even under prompt injection, and being able to prove that guarantee.

So what Security must be enforced in deterministic code, not in the prompt — and verified by tests.

03 Security Principle

The model proposes; deterministic code disposes

Zero-trust

The model is never trusted for authorization — verification is enforced in code it cannot influence.

Defense in depth

Prompt rules, then the code gate, then a tool-level assertion. Three independent layers.

Proven, not assumed

Adversarial evaluations confirm injection never bypasses the gate.

04 Transfer Flow

Every transfer clears two factors — identity, then explicit authorization

So what Identity (security question) and authorization (confirming the exact action) are separate controls — and infeasible transfers are rejected up front.

05 State-Aware Security

Verification is an explicit state machine — time-limited, single-use, and lockable

Time-to-live

A verified session expires after two minutes.

Single-use

Each transfer needs its own fresh verification.

Lockout

Three wrong answers lock the session and raise an alert.

06 Engineering

Recognised design patterns keep the system secure and maintainable

Finite state machine

Verification lifecycle as explicit states and legal transitions.

Policy enforcement point

One interceptor gates every sensitive tool.

Human-in-the-loop

Explicit approval of the exact action before money moves.

Defense in depth

Prompt, code gate, tool assertion, redaction, audit.

Risk-based access

Read-only is frictionless; money movement requires step-up.

Adapter (thin tools)

Tools wrap the service layer where the real logic lives.

Single source of config

Every tunable in one env-overridable module.

Graceful degradation

Retries on transient HTTP and empty model responses.

Audit trail

Security events and completed transfers — without secrets.

07 Observability

Every interaction is one trace — with security events attached

Security metrics

Failed verifications, unauthorized transfer attempts, lockouts, state inconsistencies.

Agent metrics

Tool failures, loop detection, latency — time-to-first-token and multi-step.

LLM metrics

Token usage per session, cost — input vs output, audio vs text.

So what Any alert links straight to the exact conversation that caused it — with PII redacted and secrets never logged.

09 Robustness

Hardened with banking controls and graceful failure

  • Retry on failure — transient HTTP errors and empty model responses
  • Transaction limit — per-transfer cap
  • Up-front validation — infeasible transfers rejected before confirmation
  • Lockout — brute-force protection on the security answer
  • PII redaction — cards and SSNs masked before the model
  • Audit trail — completed transfers and security events
  • User isolation — identity from the session, never the model
  • Data minimization — out-of-band verification (design)

Single agent, not multi-agent

One tight security flow; multi-agent would be over-engineering.

Plain dicts, not custom Pydantic

Idiomatic for ADK tools and JSON session state; ADK validates inputs for us.

Reliable tool calling, unlike native-audio in preview.
